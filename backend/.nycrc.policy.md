# Coverage threshold policy

**Companion file to `backend/.nycrc.json`.**

The threshold values in `.nycrc.json` are a **ratchet**, not an aspiration. They
must never be lowered. They are raised incrementally as coverage improves.

## Current ratchet (2026-04-30)

| Metric       | Floor | Master measurement (2026-04-27, post PR #99) |
| ------------ | ----- | -------------------------------------------- |
| `branches`   | 60    | 60.86                                        |
| `lines`      | 69    | 69.98                                        |
| `functions`  | 76    | 76.13                                        |
| `statements` | 70    | 70.05                                        |

Each floor sits at the integer floor of the most recent measured coverage on
master — no slack. A PR that lowers any metric by even 0.1pp will trip the gate.
This is intentional.

## Why these values, why not the originally-aspirational 85/80/80/85?

The originally-aspirational values in `nycrc.json` blocked all PRs with no path
to compliance. The ratchet replaces aspiration with empirical reality: "today's
coverage is the floor; new code must maintain or improve it." Aspirational
targets are tracked separately as a beads issue.

## How to raise the ratchet

1. Run the full coverage suite: `cd backend && npm run test:coverage`.
2. Read the resulting integer floor for each metric.
3. Edit `.nycrc.json` to set each metric to the new floor (round DOWN — never
   up).
4. Update the table above with the new measurement and date.
5. Commit with `chore(coverage): raise ratchet to N/M/O/P`.

## How NOT to bypass the gate

- Never set `check-coverage: false`.
- Never lower a threshold to admit a regression.
- Never use `--no-verify` to push past the gate.
- If a PR can't meet the floor, the PR adds tests until it can.
