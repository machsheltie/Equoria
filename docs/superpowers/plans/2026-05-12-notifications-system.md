# Notifications System Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the broken JSONB-based game notification storage with a proper relational `Notification` table, add missing notification events (competition stat gains, foal births, marketplace sales), and fix the frontend stale-cache issue so the bell badge updates promptly.

**Architecture:** A `Notification` model is added to the Prisma schema with `userId` → `User` foreign key (CASCADE delete) and two indexes for fast unread-count queries. A thin `createNotification(userId, type, payload)` utility in `backend/utils/` wraps the atomic DB insert with a swallowed-error outer try/catch so notification failures never block game actions. Every game-event handler imports and calls this utility after its main action commits.

**Tech Stack:** Prisma ORM, PostgreSQL, Express.js (ES Modules `.mjs`), React Query (`@tanstack/react-query`), TypeScript (frontend), Jest + supertest (backend tests)

  ```prisma

  notifications             Notification[]
  ```

  Then, after the User model's closing brace, add the new model:

  ```prisma
  model Notification {
    id        String   @id @default(cuid())
    userId    String
    user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
    type      String
    isRead    Boolean  @default(false)
    payload   Json
    createdAt DateTime @default(now())

    @@index([userId, isRead])
    @@index([userId, createdAt(sort: Desc)])
  }
  ```

  The `@@index([userId, isRead])` index powers the fast unread-count query. The `@@index([userId, createdAt(sort: Desc)])` index powers the ordered fetch for the notifications list.
