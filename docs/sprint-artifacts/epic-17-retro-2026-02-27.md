## Retrospective

### What Went Well

**The hybrid OnboardingSpotlight architecture was the right call.**

The three-way architecture debate from Epic 16 retro (Charlie's data-driven engine + Alice's visual spotlight + Dana's `data-onboarding-target` attribute bridge) converged cleanly into a single component that handles all three concerns:

- **Data layer**: `ONBOARDING_STEPS` array in `OnboardingSpotlight.tsx` — structured step definitions with `route`, `highlightTarget`, and `message` fields
- **Visual layer**: Fixed pulsing ring via `getBoundingClientRect()` + anchored tooltip card, `@keyframes onboarding-ring-pulse` using `--celestial-primary`
- **Bridge layer**: `data-onboarding-target` attributes on 10 key interactive elements across 10 pages/components

The **floating guide chip** fallback (shown when the user is on the wrong route) was a smart addition — the tutorial does not break when a player navigates away from the expected step; it gently redirects them with a "Go to [Location] →" button.

**`advanceOnboarding` backend is minimal and correct.**

The JSONB-merge pattern (`{ ...currentSettings, onboardingStep: newStep }`) is consistent with how `User.settings` is managed throughout the codebase. Auto-completion at step 10 is handled in one guard clause. Three test cases cover all meaningful states.

**Z-index token migration was mechanical and thorough.**

`mcp__serena__replace_content` with `allow_multiple_occurrences: true` made 55+ file updates tractable. The semantic mapping was well-specified upfront:

| Raw class         | Semantic         | Token               | Value |
| ----------------- | ---------------- | ------------------- | ----- |
| `z-50`            | Modals           | `--z-modal`         | 70    |
| `z-40`            | Sticky bars      | `--z-dropdown`      | 40    |
| `z-30`            | Floating actions | `z-[30]` (no token) | 30    |
| `z-20` menus      | Dropdowns        | `--z-dropdown`      | 40    |
| `z-20` close btns | Sticky UI        | `--z-sticky`        | 20    |
| `z-10`            | Content          | `--z-raised`        | 10    |

Post-migration `grep` confirmed zero raw `z-*` Tailwind classes remain. `npx vite build` clean.

**DOM ordering edge case caught and handled correctly.**

`HorseCard` badges (z-20) sit later in DOM than the gradient overlay (z-10). When both become `z-[var(--z-raised)]` = 10, badges still paint on top because of DOM order — safe. `FantasyModal` close button (z-20) appears _before_ content (z-10) in DOM, so it needs `z-[var(--z-sticky)]` (= 20) to prevent content painting over it. Both edge cases identified and handled without regression.

### What Didn't Go Well

**`rollup-plugin-visualizer` was silently broken before the session.**

The package directory existed in `frontend/node_modules/` but contained only `dist/` and a nested `node_modules/` — no `package.json`. `npx vite build` failed until `npm install` reinstalled it. Root cause: likely an interrupted earlier install. No data lost, but it added friction.

**OnboardingSpotlight `getBoundingClientRect()` will return zero rects for elements inside lazy-loaded routes on first render.**

The spotlight component uses a 100ms timeout to allow route transitions to settle, but lazy-loaded chunks that haven't mounted yet will produce a `{ width: 0, height: 0 }` rect. This is acceptable for now (the floating chip fallback activates), but a `ResizeObserver`-based approach would be more robust if the issue surfaces in QA.

**`data-onboarding-target` on the Competition enter button is inside `CompetitionDetailModal`**, not `CompetitionBrowserPage`. The modal must be open for the element to exist in the DOM. Step 9 of the spotlight will always show the floating chip (modal closed state) until the user opens a competition detail. The step message should clarify "Open a competition listing, then click Enter." — currently it just says "Enter a competition." Minor UX polish item.

## Architecture Decisions

### OnboardingSpotlight Component Contract

```typescript
// Step definition shape
interface OnboardingStep {
  step: number;
  message: string;
  route: string;
  highlightTarget: string; // matches [data-onboarding-target] value
  location: string; // human-readable location name for guide chip
}

// Rendering logic
// - Element found (correct route, DOM present): pulsing ring + tooltip
// - Element not found (wrong route or modal closed): floating guide chip
// - Step >= 10 after advance: spotlight unmounts, completedOnboarding becomes true
```

### Z-Index Token System — Final State

Tokens defined in `frontend/src/styles/tokens.css`:

```css
--z-below: -1 --z-base: 0 --z-raised: 10 /* sticky headers, content overlays */ --z-sticky: 20
  /* close buttons, floating labels */ --z-nav: 30
  /* bottom nav (z-[30] — no direct token, kept as arbitrary) */ --z-dropdown: 40
  /* menus, popovers, sticky action bars */ --z-tooltip: 50 /* tooltips */ --z-modal: 70
  /* modal backdrops and panels */ --z-above-all: 100
  /* debug overlays — FeatureFlagContext left as zIndex: 9999 */;
```
