## UX Polish Sprint (UI-1–7) — Completed 2026-02-26

This sprint ran immediately after Epic 15 and implemented the full Celestial Night visual design system.

**UI-1: Token Migration Pass**

- 16 component files migrated from raw hex/rgba to CSS custom properties
- Zero raw hex values remain in any component file
- Canonical token system: `frontend/src/styles/tokens.css` (imported in `index.css`)
- Example: `rgba(37, 99, 235, 0.1)` → `var(--celestial-primary-10)`, `rgb(212, 168, 67)` → `var(--gold-500)`

**UI-2: LocationCard Component**

- New `LocationCard.tsx` — 3-layer atmospheric card for World Hub
- Layer 1: `paintingGradient` CSS gradient as atmospheric background painting
- Layer 2: glass chrome overlay (`--glass-surface-heavy-bg`, no additional blur)
- Layer 3: hover-reactive label with `--celestial-primary` gold glow border
- React `useState` for hover (CSS custom properties can't be targeted by Tailwind JIT hover)
- Alert badge support for unread counts
- `WorldHubPage.tsx` upgraded: 9-card grid using `LocationCard` with unique atmospheric gradients

**UI-3: GlassPanel + StatBar Primitives**

- `GlassPanel.tsx` — enforces one-blur-layer rule via `noBlur` prop; 3 size variants (`sm/md/lg`) mapping to `--glass-surface-subtle/standard/heavy-bg`; polymorphic `as` prop
- `StatBar.tsx` — `role="progressbar"` + full ARIA attrs; token-driven fill (`fillColor` prop defaults to `--celestial-primary`); 3 height variants; label + track + value layout

**UI-4: Navigation Visual Polish**

- `MainNavigation.tsx` — active states converted from Tailwind gold classes to inline CSS tokens
- Desktop: `borderBottom: '2px solid var(--celestial-primary)'`
- Mobile: `borderLeft: '4px solid var(--celestial-primary)', paddingLeft: '8px'`
- `aria-current="page"` on all active items (WCAG 2.1 AA)
- Removed `text-celestial-gold`, `border-celestial-gold/50` references from active states

**UI-5: Skeleton Loading Screens**

- `SkeletonCard.tsx` — exports `SkeletonBase`, `SkeletonHorseCard`, `SkeletonLocationCard`, `SkeletonTraitBadge`
- `@keyframes skeleton-sweep` added to `index.css` — sweeps `background-position` 200% → -200%
- Animation uses `var(--skeleton-shimmer-duration)` token — zeroed to `0s` in `prefers-reduced-motion` media query (no JS required for accessibility compliance)
- All skeletons are `aria-hidden="true"` or have `role="status"` as appropriate

**UI-6: HorseCard CareStatusStrip Final Polish**

- `getCareUrgencyColor(date, warnDays, errorDays)` helper returns `--status-success/warning/error`
- Urgency thresholds: Shod warn≥7d/error≥14d, Fed warn≥1d/error≥3d, Trained warn≥7d/error≥14d
- Foaled (mare-only): informational `--text-secondary` (no urgency — just informs breeding cooldown)
- Strip labels use `--text-muted` token (was hardcoded `text-white/50`)
- `data-testid` attrs added to all date spans for future test coverage

**UI-7: Foal Dev Panel — Bonding/Stress Meters**

- All 4 progress bars across `FoalMilestoneTimeline`, `CurrentMilestonePanel`, `EnrichmentActivityPanel` updated
- Added `role="progressbar"` + `aria-valuenow/min/max` + `aria-label` to every bar (previously missing on 3 of 4)
- Fill colors: `background: 'var(--status-success)'` (was `bg-emerald-500` Tailwind class)
- Track backgrounds: `background: 'var(--bg-surface)'` (was `bg-[rgba(15,35,70,0.5)]`)
- Transitions: `var(--duration-reveal) var(--ease-out)` (was hardcoded `transition-all duration-500`)
- `getProgressColor()` in `CurrentMilestonePanel` refactored to `getProgressFill()` returning CSS token values instead of Tailwind class strings

## What Went Well

### 1. 80% Bundle Reduction in One Story (10/10)

**Technique:** `React.lazy()` + `Suspense` at route boundaries.

**Result:** Initial load went from 1.6MB to 321KB. The game now loads in ~1s on a typical connection vs ~4-5s before.

**Why it worked:** Every heavy route (Breeding, Competition, Horse Detail) was already isolated as a top-level page component. Code splitting was mechanical — wrap in `lazy()`, add `Suspense` fallback, done.

### 2. Token System Makes UI Sprint Systematic (10/10)

**Approach:** All UI-1–7 stories consumed pre-defined tokens from `tokens.css`. No colour decisions made during implementation — all colours were already specified.

**Result:** 7 UI stories completed in one session with zero design debates. Implementation was mechanical application of an already-decided design.

**Learning:** Token systems pay back their upfront cost immediately at implementation time.

### 3. `getCareUrgencyColor` — Data-Driven UX (9/10)

**Design:** Urgency colors change automatically as days pass — green today, amber next week, red when overdue. No manual state management.

**Result:** The care strip is now a genuine at-a-glance health dashboard for each horse card. Players can scan their stable and immediately see which horses need attention.

### 4. Skeleton + Reduced Motion = Accessible Loading (9/10)

**Implementation:** `--skeleton-shimmer-duration: 0s` in `prefers-reduced-motion` stops the shimmer without any JavaScript. The skeleton shapes remain (content is still loading) but there's no moving animation.

**Result:** Fully WCAG 2.1 AA compliant loading states with zero runtime JavaScript overhead.

### 5. Foal Panel ARIA Retrofit (8/10)

Three of four foal progress bars were missing `role="progressbar"` and ARIA attributes — a regression from the original implementation. UI-7 systematically fixed all four bars in one pass.

**Learning:** ARIA attributes need to be part of the initial component checklist, not added in a retrofit sprint. Added to PATTERN_LIBRARY.md accessibility patterns.

## What Didn't Go Well

### 1. Onboarding Depth (6/10)

**Problem:** 15-2 implemented the mechanics of onboarding (starter kit, flag, welcome screen) but the onboarding experience itself is thin. There's no tutorial, no guided first action, no "here's how to train your horse" flow.

**Impact:** New players still have to discover the game loop themselves. The welcome screen exists but doesn't teach.

**Recommendation:** Epic 16 should include a proper guided onboarding flow (3-step wizard: breed your first horse → assign a groom → enter your first competition).

### 2. Inventory Equip — Soft Deferral (5/10)

**Story 15-5** wired inventory equip to a "Coming soon" toast rather than a real endpoint. The backend has no inventory equip API. This is a deferred feature wearing completion clothes.

**Impact:** Players can see items in inventory but can't use them. The toast is honest but unsatisfying.

**Recommendation:** Epic 16 should include a real `POST /api/inventory/equip` endpoint and frontend flow.

### 3. Mobile Modal Z-Index (7/10)

The mobile modal z-index fix (15-6) required auditing 12 modal components to find which ones had `z-50` vs `z-[9999]`. A project-wide z-index token system would have prevented this.

**Recommendation:** Add `--z-modal: 9000`, `--z-bottom-nav: 800`, `--z-toast: 9100` to `tokens.css`. Enforce via ESLint rule.

## Critical Decisions

### Decision 1: Inventory "Coming Soon" Toast vs. Real Implementation

**Approach:** Wire inventory equip to a toast notification rather than build the backend endpoint.

**Rationale:** Backend has no inventory equip API. Building it in Epic 15 would have pulled in schema changes (inventory items, equipment slots, stat bonuses), testing, and migration work — out of scope for a polish sprint.

**Verdict:** ✅ Correct scope management. Honest placeholder is better than a broken feature.

### Decision 2: `getProgressFill()` Returns CSS Values, Not Tailwind Classes

**Approach:** Refactored `getProgressColor()` in `CurrentMilestonePanel` to return `'var(--status-success)'` instead of `'bg-emerald-500'`.

**Alternative:** Keep returning Tailwind classes, change the fill div to use `className={progressColor}`.

**Rationale:** CSS custom properties can express semantic tokens (`--status-success` adapts to dark/light mode). Tailwind utility classes cannot use design tokens without JIT config changes. Inline styles are the correct approach for token-driven dynamic values.

**Verdict:** ✅ Correct. Consistent with the entire token system approach.

### Decision 3: UX Polish Sprint Immediately After Epic 15

**Approach:** Rather than schedule UI polish as "Epic 16 story 1", complete it as an immediate follow-on sprint before declaring the project done.

**Rationale:** Design polish is easier to complete before launch when all components are fresh in memory. Post-launch polish often gets deprioritized.

**Verdict:** ✅ Correct. All 7 UI stories were completed in one session.
