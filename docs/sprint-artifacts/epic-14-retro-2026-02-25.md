## What Went Well

### 1. Multi-Stage Docker Build (10/10)

**Architecture:**

```dockerfile
# Stage 1: Build frontend
FROM node:20-alpine AS frontend-builder
WORKDIR /app/frontend
COPY frontend/ .
RUN npm ci && npx vite build

# Stage 2: Production
FROM node:20-alpine
WORKDIR /app
COPY backend/ ./backend/
COPY --from=frontend-builder /app/frontend/dist ./backend/public/
RUN cd backend && npm ci --production
CMD ["node", "backend/server.mjs"]
```

**Benefits:**

- Single Railway service (no separate frontend static host needed)
- Frontend assets served by the same Express server as the API
- Zero CORS issues in production (same origin)
- `--from=frontend-builder` keeps the final image lean (no frontend devDependencies in production image)

**Result:** ~180MB final Docker image. Clean, minimal.

### 2. Relative API URLs (9/10)

**Change:** `api-client.ts` changed from:

```ts
const baseURL = process.env.VITE_API_URL || 'http://localhost:3000';
```

to:

```ts
const baseURL = import.meta.env.VITE_API_URL ?? '';
```

**Result:** Empty string `baseURL` means all API calls use relative paths (`/api/auth/login`, `/api/horses`, etc.). In production, both frontend and backend are served from the same Railway URL — no absolute URL configuration needed.

**Benefit:** Zero-config deployment. No `VITE_API_URL` environment variable needed in Railway.

### 3. migrate deploy Idempotency (10/10)

**Pattern:** `prisma migrate deploy` only runs pending migrations. On first deploy: applies all migrations. On subsequent deploys: usually a no-op. If a migration was added: applies it.

**Result:** Zero-downtime database updates. No manual database intervention required.

**Benefit:** Developers can deploy new migrations by simply running `git push master`.

### 4. Sentry Opt-In Design (9/10)

**Pattern:** Sentry activates only when `VITE_SENTRY_DSN` is set. Without it, `initSentry()` is a no-op and `SentryErrorBoundary` renders children directly.

**Result:** Development environment is unaffected. Production can enable Sentry without code changes.

### 5. RAILWAY_SETUP.md Completeness (8/10)

The setup guide walks through every step:

1. Create Railway project
2. Add PostgreSQL plugin
3. Deploy from GitHub repo
4. Configure environment variables (with table of all required vars)
5. Verify deployment (health check endpoint)
6. Set up staging environment
7. Monitor with Sentry

A fresh developer can deploy Equoria to Railway in ~30 minutes following this guide.

## What Didn't Go Well

### 1. `npx vite build` vs `npm run build` (4/10)

**Problem:** `npm run build` in `frontend/package.json` runs `tsc && vite build`. The `tsc` step fails because of ~30 pre-existing TypeScript errors in non-critical files. This caused the Docker build to fail initially.

**Fix:** Changed Dockerfile to `npx vite build` directly (bypasses `tsc`). TypeScript checking remains in the CI `lint` job, not in the Docker build.

**Impact:** ~45 minutes of debugging. The fix is correct but the root cause (pre-existing TS errors) should be resolved in a future cleanup sprint.

**Documentation:** Added to MEMORY.md: "Use `npx vite build` directly to skip tsc (tsc runs in CI code-quality job instead)."

### 2. Bundle Size (5/10)

**Observation:** The frontend build produces a ~1.6MB JavaScript bundle. For a game, this is acceptable, but it's large for a web app.

**Analysis via bundle-stats.html:**

- Largest chunks: React DOM (~130KB), Recharts (~200KB), Radix UI components (~180KB)
- Many small chunks from the game's own components

**Recommendation:** Code-splitting for large routes (lazy-load Competition page, Breeding page, etc.) could reduce initial load time.

**Impact:** Lighthouse performance score is in the "warn" range (< 0.6) for initial load on slow connections. Not a blocker for launch.

### 3. Lighthouse CI Running Against Mock Server (6/10)

**Problem:** The Lighthouse CI job starts the Express server without a real database (no `DATABASE_URL` in CI). Some pages that require authentication fail to load content, affecting Lighthouse scores.

**Fix:** Lighthouse runs against static pages only (login, home, public pages). Game pages behind auth are not tested.

**Impact:** Accessibility score is accurate (tests static markup); performance score is slightly misleading (tests stripped-down pages).

## Critical Decisions

### Decision 1: Monolithic Architecture (Single Railway Service)

**Approach:** Express serves both the API and the React SPA from one service.

**Alternative considered:** Separate Railway services (frontend static host + backend API).

**Rationale chosen:** Simpler, fewer services to manage, zero CORS issues, lower Railway usage costs.

**Verdict:** ✅ Correct for current scale. Can always split later if API traffic requires independent scaling.

### Decision 2: migrate deploy on Every Deploy

**Approach:** Always run `prisma migrate deploy` at container startup.

**Risk:** If a bad migration is deployed, the server fails to start and Railway shows the error clearly.

**Result:** Fail-fast is better than silent data corruption.

**Verdict:** ✅ Correct. The risk of bad migrations is handled by the CI test suite.

### Decision 3: Lighthouse Thresholds (Warn Not Error)

**Approach:**

- Accessibility: error threshold 0.85 (hard fail on poor accessibility)
- Performance: warn threshold 0.6 (flag but don't block deployment)

**Rationale:** Accessibility is non-negotiable; performance can be improved iteratively without blocking releases.

**Verdict:** ✅ Correct. Accessibility first.
