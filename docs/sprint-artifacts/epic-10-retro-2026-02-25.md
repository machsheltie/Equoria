Every location page followed the same template:

1. Breadcrumb navigation (World > Service Name)
2. Header with back arrow + emoji icon + title + description
3. Tab list (My Horses + Services/Shop)
4. Tab panel content

This was established for VeterinarianPage and copied for every subsequent page.

### Groom System as Template ✅

## What Went Well

### 1. Location Page Template Reuse (10/10)

**Evidence:** VeterinarianPage was built first and then copied with minor modifications for Farrier, Feed Shop, and Tack Shop. The structural diff between pages was < 50 lines each.

**Benefit:** 4 complete, polished location pages built in a fraction of the time it would take from scratch.

### 2. Vet Booking UX (9/10)

**Pattern:** Click horse → horse expands inline showing service buttons → click service → loading spinner on button → confirmed state (green checkmark replaces spinner).

**Benefit:** Clear feedback loop. User sees exactly what's happening at each step without navigation.

### 3. Schema Fields Forward-Compatible (8/10)

**Decision:** Add `lastFarrierDate`, `hoofCondition`, `lastShod` etc. as optional fields (`?`) with sensible defaults. Existing horses without these fields work fine; new purchases update them.

**Benefit:** Migration doesn't break existing data.

### 4. Tack Shop Differentiation (8/10)

**Decision:** Tack shop cards show equipment bonus labels (`+15% Training Effectiveness`) rather than service duration. This differentiates a shop from a service business appropriately.

**Benefit:** Players understand they're buying items, not booking appointments.

### 5. GroomsPage Pattern Match (9/10)

Wrapping existing `MyGroomsDashboard` + `GroomList` in a `GroomsPage` shell followed the same two-tab pattern as `RidersPage`. Zero new component work — just a new page container.

## What Didn't Go Well

### 1. Schema Migration Required (5/10)

**Problem:** The farrier and feed shop pages required new Horse schema fields that didn't exist. This meant writing a Prisma migration mid-epic, which broke the "no DB changes" assumption for frontend sprints.

**Fix:** Migration was written and applied cleanly. No data loss or breaking changes.

**Impact:** ~30 minutes of migration setup. Not blocking, but unexpected.

**Lesson:** Check if required fields exist in schema before starting a story. Add schema changes to story prerequisites.

### 2. `useVet` Hook Initial Typing (5/10)

**Problem:** `VetService` type was defined in the hook file (`useVet.ts`) but also needed in `VeterinarianPage.tsx` which imported it. The re-export pattern caused a brief TypeScript import error.

**Fix:** Explicit `export type { VetService }` from the hook file.

**Impact:** ~15 minutes. Documented in the hook file with a comment.

### 3. Feed Shop Tab Label Ambiguity (6/10)

**Problem:** The tab was originally labeled "Services" (matching Vet/Farrier) but feed items aren't services — they're products. Changed to "Shop" mid-story.

**Impact:** Trivial rename, but it revealed an inconsistency in how tabs are labeled across the World location pages.

**Note:** Vet/Farrier use "Services"; Feed Shop/Tack Shop use "Shop." This distinction is intentional but should be documented.

## Critical Decisions

### Decision 1: Shared Location Page Template

**Approach:** Build first page (VeterinarianPage) as the template; copy for subsequent pages.

**Result:** 4 polished pages in 1 day.

**Verdict:** ✅ Correct. Template approach is the right call for repeated patterns.

### Decision 2: Inline Service Booking (Not Separate Modal)

**Approach:** Booking happens inline in the horse card expansion, not in a separate modal.

**Result:** Cleaner UX — see horses, select horse, book service in one screen.

**Verdict:** ✅ Correct. Avoids modal overhead for a simple two-step flow.

### Decision 3: Horse Ownership Validation in All Controllers

**Approach:** Every `bookAppointment` / `purchaseItem` / `bookService` / `purchaseFeed` validates that the `horseId` belongs to the authenticated user.

**Result:** No one can book services for horses they don't own.

**Verdict:** ✅ Correct. Security is non-negotiable at service booking.

## Known Issues Discovered

### Balance Deduction Not Validated Client-Side

The frontend doesn't check the user's balance before showing booking buttons. A user with $0 can click "Book Health Check" and the backend will return a 400 error.

**Recommendation:** Add balance check before enabling booking buttons (or show disabled state when insufficient funds).

### Service Data is Hardcoded in Controllers

The vet services, farrier services, and feed catalog are hardcoded arrays in the controllers rather than database-driven. This means changing service prices requires code changes and redeploy.

**Recommendation (future):** Move service definitions to a database table for admin configurability.
