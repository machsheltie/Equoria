## What Went Well

### 1. 5 Personality Types Richness (9/10)

**Design:** Trainer personalities went from 3 (Grooms) → 4 (Riders) → 5 (Trainers). The expansion creates more nuanced hiring decisions:

| Personality    | Emoji | Best For                   |
| -------------- | ----- | -------------------------- |
| focused 🎯     | 🎯    | precision disciplines      |
| encouraging 💪 | 💪    | young/nervous horses       |
| technical ⚙️   | ⚙️    | form-based disciplines     |
| competitive 🏆 | 🏆    | performance/racing         |
| patient 🌿     | 🌿    | difficult horses, dressage |

**Quote:** "Choosing a trainer personality feels like a real decision now." — Alice

### 2. TrainerCareerData Interface Design (9/10)

**Pattern:** `TrainerCareerData` is explicitly typed with all fields needed for `TrainerCareerPanel`. Components import the interface and get compile-time safety.

**Key field:** `totalSessions: number` — future sessions data; defaults to 0 until training session API is wired.

### 3. World Hub 9th Card (10/10)

**Achievement:** Adding the Trainers card to the World Hub required only 1 object addition to the `worldLocations` array. The hub grid automatically accommodated the 9th card.

**Result:** The World Hub is now truly comprehensive — all three staff types are accessible from it.

### 4. 80-Week Retirement vs 104-Week (Riders) (8/10)

**Design decision:** Trainers retire faster (80w vs 104w) because they're specialists who burn out sooner than versatile riders. This creates different long-term planning strategies for trainers vs riders.

**Game mechanic benefit:** Players must plan trainer succession more actively than rider succession.

### 5. inline Types (No Separate Type Files) (8/10)

**Decision:** Trainer types defined inline in the component/hook files rather than in separate `@/types/trainer*.ts` files. This simplifies the codebase since the type complexity is lower than Grooms (which had 7 separate type files).

**Result:** Fewer files, easier navigation for smaller type surface area.

## What Didn't Go Well

### 1. No Retirement Status Calculation Function (5/10)

**Problem:** `MyTrainersDashboard` lacks an equivalent to `calculateRiderRetirementStatus()` from the Rider system. Trainer retirement warnings are computed inline in the component rather than in a reusable function.

**Impact:** If retirement logic changes, it must be updated in multiple places.

**Recommendation (AI-13-1):** Extract `calculateTrainerRetirementStatus()` matching the Rider pattern.

### 2. 5 Personality Badges vs 4 Color Slots (5/10)

**Problem:** The personality badge system was designed for 4 personality types (matching Riders). Adding a 5th (`patient 🌿`) required adding an additional color (`emerald`) that hadn't been considered in the original design.

**Fix:** Added `emerald` color class for patient personality. No visual conflicts.

**Impact:** ~15 minutes. Documented in `TrainerPersonalityBadge.tsx`.

### 3. TrainerCareerPanel `hiredDate` Placeholder (6/10)

**Problem:** `TrainerCareerData.hiredDate` is populated with `new Date().toISOString()` as a placeholder in `MyTrainersDashboard`. The actual hire date should come from the backend.

**Fix (future):** Backend `Trainer` record includes `createdAt` which maps to `hiredDate`.

**Impact:** Shows incorrect "Hired Today" in career panel for all existing trainers until API sends real `createdAt`.

## Critical Decisions

### Decision 1: 5 Personalities (Not 4)

**Approach:** Trainers get 5 personalities to reflect the broader range of coaching styles.

**Game rationale:** A trainer who is patient with difficult horses is a distinct type from one who is technically focused on form.

**Verdict:** ✅ Correct. More personality types = more meaningful hiring decisions.

### Decision 2: 80-Week Mandatory Retirement

**Approach:** Trainers retire at 80 weeks (not 104w like Riders).

**Game rationale:** Specialist coaching is more intense; trainers "burn out" faster.

**Verdict:** ✅ Correct. Differentiated retirement timelines add strategic depth.

### Decision 3: Inline Types (No Separate Type Files)

**Approach:** Types defined near their usage rather than in separate files.

**Rationale:** Trainer type surface area is smaller than Grooms (which needed 7 type files for complex talent trees, legacy systems, etc.). YAGNI — no need to over-engineer.

**Verdict:** ✅ Correct. When to extract is a judgment call; this was the right call here.

## Known Issues Discovered

### Training Session Integration

`TrainerCareerPanel` shows `totalSessions: 0` because training sessions don't yet record which trainer was used. The backend training session service needs to capture `trainerId` and update the trainer's session count.

### Trainer Discovery Always Empty

Same issue as Rider discovery: `buildEmptyTrainerDiscoveryProfile()` generates all-unknown slots. Backend discovery service for trainers not yet implemented.
