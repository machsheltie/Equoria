## What Went Well

### 1. Concentrated Sprint Format (10/10)

**Evidence:**

- All 6 stories completed in ~1 day
- Backend APIs were already production-quality — no backend changes needed
- Frontend mock structure mirrored API response shapes closely (minimal data transformations)

**Quote:** "When the backend is already done and the mocks match the API shape, wiring is just plumbing." — Charlie

### 2. React Query as the Integration Layer (9/10)

**Pattern:** Each story created or extended a React Query hook in `/hooks/api/` rather than calling `fetch` directly in components. This gave consistent caching, loading states, and error handling for free.

**Benefits:**

- Uniform loading/error UX across all pages
- Automatic cache invalidation after mutations
- Stale-while-revalidate keeps the UI responsive

### 3. API Client Abstraction (`api-client.ts`) (9/10)

The `api-client.ts` singleton pattern — grouping all calls per domain (`authApi`, `horseApi`, `groomApi`, etc.) — made it easy to locate and update any API call. All hooks import from `api-client.ts`, not directly from `fetch`.

### 4. Zero Backend Changes (10/10)

Epic 8 didn't require a single backend file change. The backend was genuinely production-ready before this sprint began. This validated Epic 7's backend-first approach.

### 5. Pre-Existing Hook Stubs (8/10)

Several hooks had already been stubbed with correct signatures during frontend development. Wiring was often just replacing a mock return with a `useQuery` call.

## What Didn't Go Well

### 1. Auth Token Race Condition (5/10)

**Problem:** On first page load after login, the JWT token wasn't always present in the Authorization header for the initial data fetch. The `AuthContext` wasn't fully hydrated before child components mounted.

**Fix:** Added `isLoading` guard in `AuthContext` — child routes wait for auth state to resolve before first render.

**Impact:** Caused 500 errors on refresh during development, blocked Story 8-2 briefly.

### 2. CORS Configuration Gap (6/10)

**Problem:** During local development, the frontend `localhost:5173` hit the backend `localhost:3000`. Initial attempts got CORS errors until `ALLOWED_ORIGINS` was correctly set in `.env`.

**Fix:** Documented `ALLOWED_ORIGINS` in `.env.example`. Non-blocking once understood.

**Impact:** ~1 hour of debugging for a developer who hadn't seen this setup before.

### 3. Mock Data Shape Mismatches (6/10)

**Problem:** A few mock data objects used slightly different field names than the actual API response (e.g., `horse.healthRating` vs `horse.healthStatus`). Required quick fixes per story.

**Impact:** Minor delays per story. Not blocking, but added up to ~30 minutes total.

---

## Critical Decisions

### Decision 1: API Client Module Pattern

**Approach:** All API calls organized in `api-client.ts` under domain objects (`authApi.login()`, `horseApi.getAll()`, etc.). Hooks import from the client, not from raw fetch.

**Result:** Single place to update base URL, auth headers, or error handling. Clean hook files.

**Verdict:** ✅ Correct. Made the wiring sprint fast and clean.

### Decision 2: Keep All Stories P0

**Approach:** All 6 stories were P0 — none were deferred or downgraded to P1.

**Rationale:** The entire point of Epic 8 was to make the game playable. Partial completion would leave a broken prototype.

**Verdict:** ✅ Correct. The single-day completion proved the scope was achievable.

### Decision 3: React Query Cache Strategy Per Domain

**Approach:** Each domain set its own `staleTime` based on data volatility (balance: 30s, horse list: 2min, leaderboards: 5min).

**Result:** UI feels responsive without hammering the API.

**Verdict:** ✅ Correct. Documented in PATTERN_LIBRARY.md for future epics.

## Known Issues Discovered

### React Query Invalidation Patterns Need Standardization

Some mutations forget to invalidate related queries (e.g., hiring a groom doesn't always refresh the groom list immediately). This creates stale-UI moments.

**Recommendation:** Add query invalidation checklist to DoD for mutation hooks.

### Missing Error Toasts

API errors surface in the React Query error state but don't always show user-facing feedback (toast notifications). Users see broken UI silently.

**Recommendation (AI-8-1):** Standardize error handling — all mutations should call `toast.error()` in `onError`.
