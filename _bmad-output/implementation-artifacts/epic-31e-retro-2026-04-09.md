## What Went Well

1. **The highest-complexity story (31E-1b, phenotype calculation engine) delivered 65 tests with 40+ color names — fully deterministic.** The epistatic hierarchy (base color → dilutions → patterns) was encoded correctly on the first pass. No post-CR rewrites of the phenotype logic.

2. **Lethal combination filtering worked as designed — zero O/O outcomes in statistical tests.** The reroll-up-to-100-attempts strategy with heterozygous fallback correctly prevented lethal white foals while remaining computationally lightweight. The chi-squared validation for Mendelian ratios passed (Ee × Ee → ~25/50/25).

3. **Per-locus probability architecture in 31E-5 was the right call.** AC7 required <500ms for color prediction across 17 loci. Brute-force 2^17 × 2^17 enumeration (17 billion combinations) was correctly identified as infeasible. The per-locus Punnett square + Cartesian product approach delivered within budget. This is a non-obvious optimization that was correctly specified in advance.

4. **All pre-31E action items from 31D retro were completed before epic start.** Four blocking items (competitionController import fix, chi-squared flakiness fix, temperamentDrift.mjs reconciliation, 31D frontend backlog update) were all resolved. Starting 31E with a clean baseline prevented cascading issues.

5. **Pure-function service pattern held for all 6 stories.** Every 31E service (genotypeGenerationService, phenotypeCalculationService, breedingColorInheritanceService, markingGenerationService, breedingColorPredictionService) follows the pattern established in 31B/31C/31D: no Prisma imports, deterministic inputs, rng parameter for testability. All DB work in controller layer.

6. **TEA:TR was completed for 31E** — the first time in the 31-series that a formal traceability report exists at epic-level with per-test FR mapping. 7 suites × FR linkage documented. The gate decision is traceable, not just asserted.

7. **31E-5 code review improved robustness without blocking delivery.** 7 patches applied: static import (removing fragile dynamic import pattern), self-cross guard (400 for sireId===damId prevents undefined behavior), capped renormalization (guard against floating-point errors summing past 1.0), all-lethal edge case test (empty possibleColors array). None required re-architecture — all targeted.

8. **Prisma migration risk materialized and was handled.** The 31D retro flagged "Prisma migration required — unlike 31D, new fields do not already exist" as a risk. This was correct. The migration for `colorGenotype` and `phenotype` JSONB fields ran cleanly against both dev and test DBs as planned.

## What Didn't Go Well

1. **TEA:ATDD was not confirmed for 31E-1a through 31E-4.** The traceability report exists (TEA:TR ran), but there is no artifact evidence that ATDD or RV steps ran before and after each story. The 31F retro explicitly called out TEA being mandated from 31E-5 onwards — implying 31E-1a through 31E-4 also lacked ATDD. This is the same gap that caused 31F's idempotency issue to be caught in review rather than acceptance testing.

2. **Only one story artifact exists for 31E (31E-5).** Stories 31E-1a through 31E-4 have no implementation artifact files in `_bmad-output/implementation-artifacts/`. This means dev notes, lessons learned, and struggles from those stories are not preserved. If similar patterns recur in later epics, we have no record to consult.

3. **lint-staged + Windows stash pop failures recurred through multiple 31E stories.** The 31E-5 dev notes list `rollup.win32-x64-msvc.node` lock file stash pop failures as a known issue from prior 31E stories. This is a persistent friction point that has now appeared across at least 31D and 31E stories without a permanent fix.

4. **Route ordering gotcha appeared in 31E-4 (specific routes before /:id catch-all).** This is listed in 31E-5 dev notes as a "previous story learning" — meaning at least one 31E-4 story had a route ordering bug. This is not a first occurrence; it has appeared in multiple epics.

5. **Frontend integration gap continues to compound.** Epic 31E delivers genetics, phenotype, and color prediction to the API — but no frontend surface. Players still cannot see their horse's coat color genetics, markings, or breeding prediction in the game UI. Five consecutive backend epics with zero frontend exposure.

6. **Performance NFRs for GET /genetics and GET /color are only partially verified.** The TEA:TR advisory items note that <300ms and <200ms constraints are not testable at unit-test level (no DB in unit tests). No integration performance test exists for these endpoints.

## Key Insights

1. **Story artifact preservation is a process gap, not a judgment call.** Of 6 stories, only 31E-5 has a saved implementation artifact. The dev notes from 31E-5 include "previous story learnings" that reference issues from 31E-3 and 31E-4 — which means those learnings were documented at the story level but the stories themselves weren't saved to the artifacts folder. Future epics: story files should be moved to `implementation-artifacts/` upon completion, not only created in the sprint folder.

2. **The per-locus probability pattern is a reusable algorithm.** The approach in 31E-5 (per-locus Punnett square → Cartesian product → phenotype aggregation) is general. Any future system requiring "what are the possible offspring outcomes across independent loci?" (e.g., trait genetics, behavioral genetics) can use the same pattern. Add to the Pattern Library.

3. **"Previous story learnings" in dev notes signals absent institutional memory.** The 31E-5 dev notes list 7 patterns from 31E-3 and 31E-4 that should not have been learned again within the same epic — JSONB type guard, route ordering, lint catch block syntax, rejectPollutedRequest. Each of these either exists in CONTRIBUTING.md or should be added to prevent re-discovery each story.

4. **Lethal filtering is a player-facing communication design question, not just a backend filter.** The implementation correctly prevents O/O foals via reroll. But the player never sees this happening — there's no UI signal that "this breeding pair has lethal white risk" or "your foal's genotype was adjusted." This is a frontend design debt that will need UX thought when color genetics is surfaced.

5. **TEA:TR is now the minimum quality gate — not optional.** The traceability report confirmed 10/10 FR coverage with per-test linkage. This level of traceability takes time but caught the advisory FR-33/FR-34 performance NFR gap that would otherwise be invisible. TR should remain mandatory.

## Significant Discoveries

**Per-locus probability as a general genetics pattern.** The 31E-5 solution to the performance constraint is not coat-color-specific — it applies to any multi-locus independent Mendelian system. Future genetics work (trait inheritance, behavioral genetics) should adopt the same pattern rather than re-solving the computational complexity problem.

**Artifact preservation gap.** Of 6 stories completed, only 31E-5 preserved a story file in `implementation-artifacts/`. The 31E-5 dev notes reference issues from 31E-3 and 31E-4 — confirming those stories had lessons worth recording, but they're not accessible to future developers. This is a systemic process gap, not a one-time miss.

**Coat color phenotype complexity was correctly forecast.** The 31D retro flagged "Coat color genetics significantly more complex than conformation/gait/temperament." This proved accurate — 31E-1b (phenotype engine) alone required 65 tests to cover all dilution and pattern combinations. The team navigated this successfully, but the complexity level validates the decision to give it a full epic rather than folding it into a larger epic.
