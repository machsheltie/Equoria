# Story 21S-8: Require Admin Role on `POST /api/memory/cleanup`

**Epic:** 21S - Beta Readiness Gap Closure
**Priority:** P1
**Status:** done
**Source:** `docs/sprint-change-proposal-2026-04-16-beta-readiness-gap-fixes.md` — Change G / Finding P1-9
**Owner:** BackendSpecialistAgent + SecurityArchitect
**Approach chosen:** Option A (admin role guard). Kept the HTTP surface for legitimate ops workflows while rejecting every non-admin session.

## Problem

`backend/modules/labs/routes/memoryManagementRoutes.mjs` registered `POST /api/memory/cleanup` and `POST /api/memory/gc` behind `authenticateToken` only — **any authenticated beta tester** could trigger `memoryManager.cleanupAllResources()` or force a process-wide garbage collection, affecting operational state for everyone.

## Acceptance Criteria

- [x] AC-1: `requireRole('admin')` applied to `POST /api/memory/cleanup`.
- [x] AC-2: Same guard applied to the sibling destructive route `POST /api/memory/gc` — the story explicitly allows covering adjacent operational routes.
- [x] AC-3: Integration test proves non-admin users receive 403 and admin users pass through (200 on cleanup; 200 or 400 on gc since Jest doesn't start Node with `--expose-gc`, and the 400 response originates from the controller body — proof the guard passed).
- [x] AC-4: Observability GET routes (`/status`, `/resources`, `/health`, the resource analytics GETs) remain authenticated-only. The story did not require gating those and they do not mutate state.

## Collateral fix — `requireRole` was previously non-functional

Discovered during RED-phase debugging: the JWT payload generated by
`createTokenPair` in `backend/utils/tokenRotationService.mjs` carries only
`{ userId, type, jti, iat }`. `authenticateToken` reads the decoded token
into `req.user` verbatim, so `req.user.role` was **always undefined** — meaning
every existing `requireRole('admin')` call site (notably `forumRoutes.pinThread`)
was silently rejecting every user, including real admins.

Fix: `requireRole` now lazy-looks-up the role from the database when
`req.user.role` is not already present. This keeps `authenticateToken`'s
hot path DB-free (one fewer query per authenticated request) while making
`requireRole` actually enforce roles — only the admin-guarded endpoints
pay the lookup cost.

This is a net security improvement: previously admin surfaces rejected
everyone (fail-closed but for the wrong reason); now they correctly
accept admins.

## Dev Agent Record

### Completion Notes

- `POST /api/memory/gc` was also gated. It triggers `global.gc()` process-wide; same blast radius as `/cleanup` from an ops-hygiene perspective.
- The `/gc` admin-path test accepts status 200 OR 400 because Jest doesn't start the Node process with `--expose-gc`, so the controller responds 400 ("Garbage collection not exposed"). Either outcome proves the guard passed — a non-admin caller short-circuits with 403 before reaching that branch.
- The `requireRole` lazy lookup adds ~1 DB query per admin-guarded request. Only two routes are currently admin-guarded (`/api/memory/cleanup`, `/api/memory/gc`, `/api/forum/threads/:id/pin`), so this is effectively free at beta scale.
- Non-admin tests needed `x-test-skip-csrf: true` header — the memory routes live on `authRouter` which applies `applyCsrfProtection`, and under `JEST_WORKER_ID` the backend honors the bypass (production/beta refuses it per Story 21S-2's hardening).
