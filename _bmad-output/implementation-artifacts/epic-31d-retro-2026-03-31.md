## What Went Well

1. **No Prisma migration needed** — The `temperament String?` field already existed in `schema.prisma` (line 118) and `horseModel.mjs` already destructured and persisted it. 31D-1 started clean with zero schema work, setting a fast pace for the whole epic.

2. **First existing-system integrations went smoothly** — 31D-2, 31D-3, and 31D-4 were the first stories in the 31-series to modify existing game systems (training, competition, groom) rather than adding standalone features. All three completed without regression. The careful isolation discipline (temperament modifiers stacked multiplicatively after trait effects, not mixed into trait system) prevented cascading failures.

3. **Zero-deferral maintained across all 5 stories** — Every FR delivered. No "we'll come back to this" decisions.

4. **`temperamentService.mjs` grew incrementally without debt** — The service file was created in 31D-1 with 2 exports and grew to 8 exports by 31D-4. Each story added only what it needed. No pre-emptive abstractions or bloat.

5. **`_any` sentinel key for Calm/Steady** — The design decision to use `_any` in `TEMPERAMENT_GROOM_SYNERGY` for universal personality bonuses was clean and required no special-casing outside the function itself. 31D-4 code review further hardened this with a `CANONICAL_GROOM_PERSONALITIES` set to prevent non-canonical personalities from receiving the universal bonus.

6. **31D-3 passed on first run with zero debugging** — 30 tests for competition modifiers, all passing immediately. Story spec with exact math examples (`Math.random = 0.5 → zero luck`) made the test values unambiguous.

7. **Code reviews improved robustness without blocking delivery** — 31D-4 review added `.trim()` normalization and `CANONICAL_GROOM_PERSONALITIES` validation. 31D-5 review added explicit `@jest/globals` imports and a non-empty strings assertion. Both were small, targeted improvements. No High or Medium issues.

8. **Pre-31D cleanup sprint paid off** — All 5 cleanup items completed before 31D started. Breed data enriched from `BreedData/` samples, seed mismatch fixed, flaky benchmark resolved, `nextActionsController` failures addressed. Starting 31D with a clean baseline made every regression check trustworthy.

9. **`processGroomingSession` wiring in 31D-4 was clean** — Renaming `_groomId` to `groomId`, adding groom personality fetch, and wiring to `calculateBondingEffects` were exactly as specified. No unexpected side effects on 43 existing groomBondingSystem tests.

## What Didn't Go Well

1. **`temperamentDrift.mjs` is unresolved legacy** — This utility defines a 6-type temperament system (Calm, Spirited, Nervous, Aggressive, Docile, Unpredictable) that is incompatible with our canonical 11-type system. It was correctly flagged in 31D-1 dev notes ("do not touch") and left alone, but it remains a silent landmine. Any future developer unfamiliar with this doc might import from it.

2. **`competitionController.mjs` import mismatch left open** — `competitionController.mjs` imports `calculateCompetitionScore` from `competitionLogic.mjs` instead of the canonical `competitionScore.mjs` (which we modified in 31D-3). This means the temperament modifier is NOT applied through the competition controller path. It was explicitly scoped out in 31D-3 dev notes, but it means the feature is incomplete in production competition flows despite passing all tests.

3. **No frontend integration — again** — Fourth consecutive backend epic with zero frontend surface. The horse detail page, training result responses, and competition results don't expose temperament data to players. `temperamentEffects` field is now in training responses but there's no UI for it.

4. **Two Groom personality fields caused confusion** — The Groom model has both `personality` (the canonical `gentle|energetic|patient|strict` field used by 31D-4) and `groomPersonality` (a separate epigenetic/trait field with different values). The 31D-4 dev notes called this out as CRITICAL but it should be consolidated at the schema level in a future story.

5. **Chi-squared flakiness carried from 31D-1** — The statistical distribution tests in `temperamentAssignment.test.mjs` had chi-squared flakiness noted by 31D-2 as "pre-existing unrelated to this story." It was carried again without resolution. This is now three consecutive retros where a flaky test has been noted and not fixed.

---

## Key Insights

1. **Existing-system integration stories need explicit scope boundaries** — 31D-3's "pre-existing mismatch is out-of-scope" note was correct but created a silent gap: tests pass, but the feature doesn't fire in production. Future stories modifying existing systems should include a "validate production path" task, not just unit tests.

2. **The `_any` sentinel pattern is a precedent** — Other game systems may need universal modifiers (e.g., a trait that applies to all disciplines). The `_any` key + canonical set validation pattern from 31D-4 should be recorded in the Pattern Library as the standard approach for universal selectors in lookup tables.

3. **Test math specification in dev notes = zero debugging** — Stories 31D-2, 31D-3, and 31D-4 all included exact numeric examples in dev notes (e.g., `Lazy horse base=5 → trait → temperament → min(1)`). This pattern has now proven itself across multiple stories: when the expected values are in the spec, tests pass first run.

4. **Service file growth needs an ownership check** — `temperamentService.mjs` now exports 8 symbols across 4 distinct concerns (assignment, training modifiers, competition modifiers, groom synergy). It's still cohesive (all temperament concerns) but should be reviewed at the start of 31E to decide if any split is warranted before it becomes unwieldy.

5. **Frontend integration debt compounds** — Four backend epics without UI exposure means players have never experienced conformation, gait, or temperament in the actual game. The sooner this is surfaced, the fewer API contract surprises there will be.
