## What Went Well

1. **Story consolidation saved effort** — 31C-2 (Gait Breeding Inheritance) was identified as redundant since FR-15 and FR-16 were both fully implemented within 31C-1's `generateInheritedGaitScores()`. Marked done without separate implementation, saving an entire story cycle.

2. **Zero debugging cycles** — All tests passed on first run for both 31C-1 and 31C-3. Story specs with exact formulas continue to eliminate ambiguity.

3. **Clean code reviews** — Both reviews returned 0 High/Medium issues. The 31B patterns (service module, route placement, test mock setup) transferred without adaptation.

4. **Pattern reuse at scale** — `normalRandom`, `clampScore` from conformationService reused directly. Route middleware chain (`queryRateLimiter`, `validateHorseId`, `requireOwnership`) copy-pasted from conformation endpoints. Test file structure copied from `conformationApiEndpoints.test.mjs`.

5. **First Prisma migration since 31A went smoothly** — `gaitScores Json?` field added, migration deployed to both dev and test DBs. Known Windows EPERM issue on DLL rename was expected and handled.

6. **Attention to detail on execution** — Per Heirr: "This time around we stuck to the plan. Earlier epics would say one thing and fail to implement it the way we discussed or defer and never get back to it."

---

## What Didn't Go Well

1. **NFR-05 correlation threshold deviation** — Spec said `r > 0.3`, implementation uses `r > 0.1`. Accepted as pragmatic given the subtle conformation bonus (`±1.5` typical), but it's technically a spec deviation. Spec should be updated to match reality.

2. **Pre-existing test failures unaddressed** — `nextActionsController.test.mjs` has 5 failing tests (cooldown/competition logic) that have been failing since before 31C. Nobody owned fixing them, and they've been silently ignored across multiple epics.

3. **No frontend integration** — Three epics of backend systems (31A, 31B, 31C) completed with zero frontend surface. The horse detail page doesn't display conformation or gait data. API client (`api-client.ts`) has no calls to the new endpoints. Risk of discovering API contract issues late.

---

## Key Insights

1. **Story consolidation is a valid outcome** — When a story's FRs are already covered by a previous story, marking it done without separate implementation is the right call. This avoids make-work while maintaining traceability.

2. **Pattern reuse has compounding returns** — By epic 31C, new features are essentially "fill in the template." The service module → controller → route → test → swagger pipeline is now muscle memory. This will accelerate 31D significantly.

3. **Frontend integration must be tracked proactively** — Heirr's observation that we could "get a full head of steam up on backend and then forget to implement it on the frontend" is a systemic risk. Backend epics need a visible frontend checklist.

4. **Tech debt carried across retros becomes permanent** — "Lusitano Real Data," "Dev Seed Breed Mismatch," and "Flaky Perf Benchmark" have been carried for two retros. Heirr: "That's the type of thing we forget to ever go back and address." Decision: fix all carried debt before starting 31D.

## Significant Discoveries

**Story consolidation as a practice** — 31C-2 being marked done without implementation was the first time we consolidated stories. This should be a recognized practice going forward: during story creation, if a previous story already covered the FRs, mark it done with a comment explaining the consolidation rather than creating make-work.

**Frontend integration gap is systemic** — Three backend epics with zero frontend surface is a planning failure, not an implementation failure. The fix is structural: track frontend work per backend epic and ensure it's visible in sprint status.
