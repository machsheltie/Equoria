## What Went Well

1. **Code review caught three non-trivial concurrency and performance issues in 31F-3 before production** — idempotency gap (show re-executable, points farmable), N+1 groomAssignment query in Promise.all, stale horse data read outside `$transaction`. All three were real production bugs. The fixes (status flag + atomic update in $transaction, batch query with findMany+Map, re-fetch inside $transaction) are now documented as conventions in CONTRIBUTING.md.

2. **Auth chain investigation prevented a wasted fix** — The "missing auth middleware" on POST /execute and GET /titles was initially flagged HIGH. Tracing the full mount chain (app.mjs → authRouter.use(authenticateToken) → competitionRoutes → conformationShowRoutes) confirmed routes were already protected. Correctly dismissed as false positive. Lesson: always check the full mount chain before flagging missing auth.

3. **Surgical service modification pattern worked cleanly (31F-1)** — The existing conformationShowService.mjs had the right structure; work was targeted fixes only (arithmetic mean, remove random factor, handler score map, bond score fix). No rewrites, no bloat. Fastest story of the epic.

4. **100% FR delivery, 0 production incidents** — Three stories, 8 FRs, zero incidents. The conformation show system is fully functional end-to-end.

5. **Schema re-use where possible** — 31F-1 and 31F-2 required zero Prisma migrations. Only 31F-3 needed new fields, and these were applied correctly (with the db-execute bypass as a workaround for drift).

## What Didn't Go Well

1. **TEA was completely skipped for the entire epic** — ATDD was not run before any story. TA and RV were not run after. TR was not run at epic completion. The canonical workflow mandates TEA at steps 3, 5, and 6 of every story loop. Heirr has made this non-negotiable going forward. The idempotency gap in 31F-3 is a direct consequence — a failing acceptance test before coding would have caught it at story creation, not after implementation.

2. **31F-2 needed two code review passes** — First review caught: wrong HTTP status (404 vs required 400), dead `validationResult` import, express-validator branch untestable. Second review caught: missing `.trim()` on className. Both passes found real issues. TEA:ATDD before dev and TEA:RV after would have caught most of these before review.

3. **`prisma migrate dev` blocked by drift — third occurrence** — The migration history has now drifted from the actual schema for the third time. 31F-3 had to bypass with `prisma db execute` SQL directly. This creates silent discrepancy between migration files and actual DB state. Must be resolved before Railway production deploy.

4. **Test co-location rule violated again** — 31F-3 initially placed tests in `backend/__tests__/` (root cross-cutting folder) instead of `backend/modules/competition/__tests__/` per CONTRIBUTING.md (established in Epic 21-1). ## Key Insights

1. **TEA is a delivery commitment, not an optional enhancement.** Three stories without ATDD means we shipped three times without acceptance tests written first. The pattern "code first, review catches issues" is slower and riskier than "ATDD first, code passes tests, review confirms quality." Heirr has mandated TEA from 31E-5 onwards — enforcing this at story creation, not retrospective.

2. **Idempotency must be first-class in any execution endpoint.** Any "execute show / run batch / distribute rewards" endpoint can be exploited without a status guard. The pattern is now in CONTRIBUTING.md: check `status === 'completed'` before executing; update atomically in `$transaction`.

3. **N+1 in Promise.all is an anti-pattern.** `entries.map(async entry => prisma.X.findFirst(...))` looks clean but is N+1. The fix pattern (batch `findMany` + Map by ID) is now in CONTRIBUTING.md.

4. **Stale reads in accumulation transactions are a race condition.** Re-fetch current values inside `$transaction` before computing new ones. Pattern documented.

5. **Migration drift compounds and is becoming a production blocker.** Three `prisma db execute` bypasses have widened the gap between migration history and actual schema. This must be resolved as a dedicated pre-31G task (or equivalent) before Railway deploy.

6. **Auth chain understanding is a prerequisite for security reviews.** Trace the full mount hierarchy (app.mjs → authRouter → sub-router) before flagging missing middleware.

## Discoveries

**Idempotency gap class of bug** — Any endpoint that executes state-altering batch operations (show execution, foal development evaluation, reward distribution) is vulnerable to repeated execution unless explicitly guarded with a status flag. The guard must be set atomically inside the same `$transaction` as the writes. This is now a first-class convention.

**Auth chain false positive pattern** — The authRouter-level `authenticateToken` middleware protects all routes mounted on it, including sub-routers. Reviewers must trace the full mount chain before flagging per-route auth as missing. The conformationShowRoutes sub-router is mounted on `competitionRoutes`, which is mounted on `authRouter` — fully protected.

**Test file location** — `backend/__tests__/` is for cross-cutting integration tests only. Module-level tests belong at `backend/modules/<domain>/__tests__/`. This rule is in CONTRIBUTING.md and was violated in 31F-3. Caught in review, corrected.
